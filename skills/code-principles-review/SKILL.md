# Code Principles Review

Post-production review of recently modified code focused on software design principles and library/component reuse. Complements the `simplifier` skill (which targets clarity/readability) by targeting architectural and reuse concerns.

## When to Run

- After completing a feature, refactor, or significant code change
- When the user explicitly asks for a principles review
- NOT for formatting, naming, or clarity issues (defer to `simplifier` and `coding-convention`)

## Procedure

### 1. Scope the review

Identify recently modified files via `git diff --name-only` (staged + unstaged) or the files touched in the current session. Focus exclusively on new or changed code — do not review untouched code unless it directly interacts with the changes.

### 2. Detect the project's ecosystem

Before reviewing for reinvented-wheel patterns, run these discovery steps:

- Read `package.json` (dependencies + devDependencies) to map installed packages
- Check `components/ui/` or equivalent to identify the component library (shadcn, Radix, MUI, etc.)
- Note any project-specific shared utilities (`lib/`, `utils/`, `shared/`, `common/`)

### 3. Review against principles

Walk through each modified file. For each, apply the checklist in [references/checklist.md](references/checklist.md) across four dimensions:

1. **DRY** — duplicated logic, repeated patterns, parallel conditionals
2. **SOLID** — responsibility bloat, hardcoded variants, fat interfaces, tight coupling
3. **KISS** — premature abstractions, unnecessary indirection, over-engineered patterns
4. **Library/Component reuse** — custom implementations where an installed or widely-adopted package already solves the problem; custom UI where the project's component library provides the primitive

### 4. Cross-reference the codebase

For each suspected DRY or reuse violation, search the codebase (`Grep`/`Glob`) to confirm:

- Whether similar logic already exists elsewhere (duplication)
- Whether the project already installs or wraps the relevant library
- Whether a shared component/hook already handles the pattern

### 5. Report findings

For each finding, provide:

- **Principle violated** (DRY / S / O / L / I / D / KISS / Reuse)
- **File and location**
- **What's wrong** (one sentence)
- **Suggested fix** (concrete, minimal)

Group findings by severity:
- **Must fix** — clear principle violations that will cause maintenance pain
- **Should fix** — improvements with real benefits but lower urgency
- **Consider** — minor suggestions, taste-level

If no findings, state it — do not invent issues.

### 6. Apply fixes

After presenting the report, apply the "Must fix" items directly. For "Should fix" items, ask the user before proceeding.

## Examples

### Reuse — Custom type guard replaced by Zod safeParse

```ts
// ❌ equipment-types.utils.ts — hand-rolled type guard
function isEquipmentTypeColumn(value: string): value is EquipmentTypeColumn {
  return EQUIPMENT_TYPE_COLUMNS.some((column) => column === value);
}

// usage
if (sortBy && isEquipmentTypeColumn(sortBy)) {
  return { [sortBy]: sortDir ?? 'asc' };
}
```

`zEquipmentTypeScalarFieldEnumEnum` (generated by Prisma/Zod) already does the same thing AND returns the parsed type.

```ts
// ✅ after — use the generated Zod enum
if (sortBy) {
  const parsed = zEquipmentTypeScalarFieldEnumEnum.safeParse(sortBy);
  if (parsed.success) {
    return { [parsed.data]: sortDir ?? 'asc' };
  }
}
```

**Finding**: Must fix — Reuse. Custom type guard reimplements what an already-generated Zod enum provides.
**Fix**: Replace with `.safeParse()` — delete the custom guard + its `EQUIPMENT_TYPE_COLUMNS` import.

## Gotchas

- A dependency listed in `package.json` but never imported is a strong signal of reinvention — grep for it before flagging
- Do not flag standard library usage as "reinventing the wheel" (e.g., using `Array.prototype.filter` instead of lodash is fine)
- SOLID applies differently to React components vs. backend services — a component combining render + a simple hook is not necessarily violating SRP
- Do not recommend adding new dependencies to the project; only recommend using already-installed packages or native APIs
- Avoid flagging single-use helpers as DRY violations — DRY applies when there are 2+ actual duplications, not hypothetical future ones
