# Aggregated Report Format

The final output the orchestrator returns to the user. Phase 1 summary + three review sections from three subagents, unified header, single recommendation list at the bottom.

## Structure

```
# PR Review — <branch> vs <base>

**Scope:** <N> files changed (<X> additions, <Y> deletions)
**Subagents:** code-review (quality) + architecture-review (cross-boundary) + regression-checker (blast radius)

## Phase 1 — Cleanup

Simplifier applied <N> edits across <M> files. Summary of the most impactful cleanups:
- <file:line> — <one-line description>
- ...

(Or: "Skipped — colleague branch mode, simplifier not applied.")

| **Total**    | **X**    | **Y**   | **Z** |

**Top themes:** <2-3 sentences naming the dominant patterns: e.g., "Three files reimplement Zod schemas already generated by Prisma. Two changed exports have unhandled callers in `apps/web/`. New `@repo/date-format` package overlaps with existing `@repo/i18n/dates`.">

Grouped by concern, not by file.
```

## Aggregation rules

1. **Sort within each section** by severity (critical → warning → nit), then by file path alphabetically.
2. **Numbered actions are global,** not per-section. Critical regressions and critical quality issues compete on the same priority list.
3. **Empty sections are kept** with a one-line message ("No quality issues found." or "No regressions detected."). Empty sections are signal.
4. **Subagent failure** → add a `## Subagent failures` section before "Recommended next actions" and degrade gracefully:

   ```
   ## Subagent failures
   - **Blast-radius check:** Agent timed out. Re-run manually with `regression-check` on a narrower scope.
   ```

5. **Total counts in the summary table** must match the per-section findings — if they don't, recount before emitting.

## What NOT to include

- Raw diff blocks. The user already has `git diff`.
- File contents quoted at length. Reference `file:line` instead.
- Praise or filler ("Great PR overall!"). Findings only.
- Speculation about intent. Stick to what the diff shows.
- Recommendations to add new dependencies (the source skills already enforce this — flag if a subagent slips up).
